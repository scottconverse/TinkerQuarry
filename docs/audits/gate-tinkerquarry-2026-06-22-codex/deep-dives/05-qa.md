## Findings

### QA-001 - Blocker - Packaged desktop app has no verified path to start or reach the KimCad engine

**Evidence:** The front-end engine client hardcodes relative `/api` as its base (`apps/ui/src/services/engineClient.ts:116`). In dev, Vite provides the only `/api` bridge: `apps/ui/vite.config.ts:46-50` proxies to `http://127.0.0.1:8765` and injects the dev session token. The Tauri production config only packages `../dist` and has no bundled resources (`apps/ui/src-tauri/tauri.conf.json:7-10`, `:33`). The Tauri runtime initializes Studio/OpenSCAD commands and plugins but no KimCad sidecar, web server, or `/api` proxy (`apps/ui/src-tauri/src/lib.rs:84-96`). Current provisioned API probes passed only because a separate engine process was already running (`GET /api/health` returned version `0.9.3`; `GET /api/model-status` returned local model present). The saved-design empty state was also empty (`GET /api/designs` -> `{"designs":[]}`), so there is no local library fallback.

**Expected:** A brand-new installed TinkerQuarry desktop launch should either start the local engine itself or present an in-product setup/retry path that gets the user to the describe -> render -> slice core flow.

**Actual:** Outside the dev server/provisioned environment, `/api/*` has no server. The UI can only show "engine isn't responding" / request-level toasts, while Save/Make it real remain unreachable until a user separately knows how to start `kimcad web`.

**Impact / blast radius:** Blocks the core UI flow for first-run desktop users: describe, design source retrieval, slicing, saved designs, options, health, and model status all depend on `/api`.

**Fix:** Package and supervise the engine as a Tauri sidecar, or make the desktop shell be launched by the Python `kimcad shell/web` host that owns the API. Add a startup contract: engine launched, token shared, `/api/health` succeeds, and failures show a one-click setup/retry path.

**Test:** Fresh-profile packaged-app smoke with no pre-running `kimcad web`: assert `/api/health` succeeds from the webview, then complete describe -> source -> render -> slice using the same binary a user installs.

### QA-002 - Critical - CLI model-down recovery tells users to run a nonexistent command, then that command starts a design run

**Evidence:** Valid CLI subcommands are `design`, `bench`, `web`, `shell`, `models`, and `bakeoff` (`packages/engine/src/kimcad/cli.py:32`, `:63`, `:92`, `:116`, `:129`, `:149`, `:160`). The model-unreachable handler prints `Run kimcad serve` (`packages/engine/src/kimcad/cli.py:633-638`), but docs use `kimcad web` (`docs/MANUAL.md:174`; `README.md:47`). Worse, `_normalize_argv` rewrites unknown non-near-match single words into a design prompt (`packages/engine/src/kimcad/cli.py:184-196`). Runtime probe with the repo venv showed `main(['serve'])` did not print help; it timed out after 14s because it entered the design path.

**Expected:** A first-run user whose local model server is down receives a valid recovery command and invalid commands fail fast.

**Actual:** The recovery command is invalid and, when followed literally, is treated as the prompt "serve", launching the slow design flow.

**Impact / blast radius:** CLI first-run/dependency-absent recovery is broken. Users can burn minutes on an accidental design job instead of starting the server or opening the web app.

**Fix:** Change the recovery text to the real path (`kimcad web`, `kimcad shell`, or the actual Ollama setup command). Add `serve` as an alias only if intended. Tighten `_normalize_argv` so known help/recovery words are not treated as prompts.

**Test:** Unit-test the model-unavailable branch text and `main(['serve'])` behavior. It should either be a supported alias or exit quickly with parser help.

### QA-003 - Major - Missing slice-profile load silently falls back to the engine default printer/material

**Evidence:** The UI loads printer/material options once and silently returns on failure (`apps/ui/src/App.tsx:942-946`). The "Make it real" call sends `printerKey || undefined` and `material || undefined` (`apps/ui/src/App.tsx:776`), which asks the engine to use defaults. The profile selector is only visible after a design exists and `enginePrinters.length > 0` (`apps/ui/src/App.tsx:2820`), but the Make it real button is enabled based on render/design state, not options-load state (`apps/ui/src/App.tsx:2907`). In the provisioned API, `/api/options` eventually returned a full catalog, but the probe took about 9.5s; a timeout or startup race leaves no visible profile state.

**Expected:** If profile options are unavailable, manufacturing should be blocked or explicitly labeled as using the engine default before producing G-code.

**Actual:** A partial startup/options failure can produce G-code for the default printer/material without showing the user which profile will be used.

**Impact / blast radius:** Manufacturing output can be for the wrong printer/material, undermining the "Make it real" commitment moment and the PRD claim that the user chooses their machine.

**Fix:** Track options loading/error state. Disable Make it real until a valid profile is loaded, or show the exact server default with a retry control and require explicit confirmation.

**Test:** Mock `engine.options()` timeout/failure in App tests and assert Make it real is disabled or surfaces a blocking profile error; verify no `/api/slice` call is made with undefined printer/material unless the UI says "engine default".

### QA-004 - Minor - Runtime shell and installer identity still say OpenSCAD Studio

**Evidence:** Tauri product/window/menu metadata still reads OpenSCAD Studio (`apps/ui/src-tauri/tauri.conf.json:3`, `:15`; `apps/ui/src-tauri/src/lib.rs:50`, `:118`; `apps/ui/src-tauri/Cargo.toml:4-5`).

**Expected:** The installed/runtime app identifies as TinkerQuarry.

**Actual:** The first desktop surface still presents the predecessor product identity.

**Impact / blast radius:** Confuses first-run trust, bug reports, screenshots, and support instructions. It does not by itself block core functionality.

**Fix:** Rename Tauri product/window/menu/package metadata and update MCP/status strings where they are user-facing.

**Test:** Packaged app smoke asserts title/menu/about metadata include TinkerQuarry and not OpenSCAD Studio.

## What's Working

- When the separately running engine is present, `GET /api/health` returns structured tool status: OpenSCAD and OrcaSlicer true, CadQuery false, with no crash.
- `GET /api/model-status` reports the local chat and vision models as running/present in the provisioned environment.
- Session-token protection is active: unauthenticated `POST /api/design` returns `403` with `{"reason":"session"}`.
- Empty saved designs return a clean `{"designs":[]}` rather than a 500.
- `/api/options` marks the known bad Elegoo Neptune 4 Max profile as `sliceable:false` with a reason, so a good UI can prevent that unsafe route.
