## Findings

### QA-1 - Major - First-run and dependency-absent native states are still not proven in isolation

**Evidence:** `docs/audits/gate-tinkerquarry-2026-06-23/walkthrough-summary.md` states that clean-state evidence is partial and that the installed-app smoke used real local app data. The native smoke script launches the app with inherited process environment only (`scripts/smoke-tauri-runtime.mjs:18-24`) and asserts the happy healthy case where `openscad === true` and `orcaslicer === true` (`scripts/smoke-tauri-runtime.mjs:71-85`). There is no `LOCALAPPDATA` / `APPDATA` / home override and no dependency-negative run.

**Why it matters:** The PRD and recovery plan treat first-run reachability, self-healing, and honest missing-tool states as release gates. A smoke that inherits the developer machine's existing app data and healthy toolchain proves native launch, not a true first install on a clean user profile.

**Blast radius:** All beta users installing on a fresh Windows account or a machine missing bundled/resolved tools. This is a release-confidence gap, not evidence that the current happy path is broken.

**Concrete fix path:** Extend `scripts/smoke-tauri-runtime.mjs` with `--isolated-profile=<dir>` that sets `LOCALAPPDATA`, `APPDATA`, `USERPROFILE`-adjacent app storage where Tauri/WebView honors it, and asserts first-run UI. Add a second mode that deliberately points engine config/tool paths at absent binaries or an empty temp bundle and asserts the UI shows the missing/degraded state instead of crashing. Run both against the installed NSIS copy.

### QA-2 - Major - Browser automation proves one desktop happy path, not the runtime matrix

**Evidence:** The committed Playwright test is a single spec, `apps/ui/e2e/manufacturing-flow.spec.ts:9-107`. It covers desktop boot, prompt/build, right rail, slice, mock connector send, and outcome. It does not cover mobile/narrow layout, failed design, failed slice, stale edit refusal, missing connector setup, hardware connector outcomes, export formats, settings library admission, keyboard traversal, or offline/dependency-missing UI.

**Why it matters:** The most failure-prone surfaces in this product are state transitions and degraded states. The happy path is now real, but release readiness still depends on catching regressions in the states a normal beta user will hit when tools are missing, slicing fails, models are absent, or the layout changes.

**Blast radius:** UI runtime confidence. A refactor could break error handling, mobile layout, or export/library flows while the current e2e suite stays green.

**Concrete fix path:** Split Playwright into tagged specs: `manufacturing-happy`, `manufacturing-errors`, `settings-libraries`, `exports`, `responsive`, `accessibility-keyboard`, and `native-smoke`. Keep the current test as the happy path, then add deterministic engine fixtures or route mocks for negative states.

### QA-3 - Major - Signature Visual Correction Loop is wired, but not runtime-proven as a correction loop

**Evidence:** The app has a bounded autonomous loop (`apps/ui/src/App.tsx:1058-1150`) and the UI reports VCL state. The e2e test only asserts that the VCL label and evidence rail are visible (`apps/ui/e2e/manufacturing-flow.spec.ts:54-57`). It does not simulate a visual issue, verify `/api/visual-review/<rid>` returns an issue, verify a correction prompt is applied, or verify best-candidate restore after regression.

**Why it matters:** VCL is a differentiating product claim. Showing a label and evidence rail proves UI plumbing, but not that the loop catches, corrects, bounds, and records visual issues in the runtime path.

**Blast radius:** Product differentiation and user trust. If visual review/correction regresses, current browser automation may not catch it.

**Concrete fix path:** Add a deterministic visual-review fixture mode to the demo engine or Playwright route mocks: first review returns `issues` with a correction prompt, second returns either `ok` or a lower readiness score. Assert the app calls the correction path, increments rounds, appends iteration log entries, preserves Undo, and restores the best candidate on regression.

### QA-4 - Minor - External library admission can admit a folder with no `.scad` file

**Evidence:** `packages/engine/src/kimcad/external_libraries.py:18` allows `.scad`, `.md`, `.txt`, `.json`, `.yaml`, and `.yml`. `file_count` increments for any allowed suffix (`external_libraries.py:82-100`), and the empty-library rejection checks only `file_count == 0` (`external_libraries.py:102-104`). A folder containing only `README.md` would be admitted even though the user-facing error promises `.scad` detection.

**Why it matters:** The Settings flow can show an admitted external library that cannot actually provide OpenSCAD modules. That creates a confusing include prefix and later render failures.

**Concrete fix path:** Track `scad_count` separately. Reject when `scad_count == 0`, keep `file_count` for total copied files, and add a unit test for a docs-only folder.

### QA-5 - Minor - Release validation script can pass without running product tests

**Evidence:** `package.json:29-30` defines `test:scripts` as a no-op message and `validate:changes` as lint + type-check + no-op script tests. It does not run UI Jest, web unit tests, engine pytest, Playwright, Tauri smoke, or cargo tests.

**Why it matters:** A maintainer can run the named validation command and get a green result even if runtime tests are broken. The detailed docs list the full proof commands, but the default validation script does not enforce them.

**Concrete fix path:** Rename this script to `validate:static` or expand it into `validate:core` that runs lint, type-check, UI Jest, web unit, engine focused/full as appropriate, and documented e2e smoke lanes. Keep a separate fast script for pre-edit checks.

### QA-6 - Minor - External-library API returns local source and sandbox paths

**Evidence:** `admit_library` records and returns `source_path` and `sandbox_path` (`packages/engine/src/kimcad/external_libraries.py:110-118`), and `/api/libraries` returns admitted records unchanged (`packages/engine/src/kimcad/webapp.py:1741-1750`).

**Why it matters:** This is a local-first app, so it is not a remote data leak by itself. But the UI and any debug export/report consuming `/api/libraries` may expose full local filesystem paths unnecessarily.

**Concrete fix path:** Keep paths in the manifest for local operation if needed, but return a redacted public shape from `/api/libraries`: name, slug, include_prefix, file_count, bytes, and maybe a basename-only source label. Add a test that the public API omits absolute paths.

### QA-7 - Nit - OpenSCAD sanitizer docstring still says only `library/` survives

**Evidence:** `packages/engine/src/kimcad/openscad_runner.py:263` says only `use/include` inside `library/` survive. The actual allowlist includes both `library/` and `external/` (`openscad_runner.py:39-42`, `openscad_runner.py:119-128`).

**Why it matters:** This is doc drift, not a runtime bug. It will mislead future security reviewers.

**Concrete fix path:** Update the docstring and blocked-message wording to say `library/` or admitted sandbox `external/`.

## What's Working

- Native packaging is real enough to build MSI/NSIS artifacts and launch the release/installed app. The smoke script verifies title, visible start surface, `ensure_engine`, authenticated `/health`, OpenSCAD, and OrcaSlicer (`scripts/smoke-tauri-runtime.mjs:61-131`).
- The main web manufacturing flow is now browser-proven through prompt, design-ready state, right rail, first-real-print caution, slice, "Ready to print", mock send, and outcome recording (`apps/ui/e2e/manufacturing-flow.spec.ts:32-107`).
- "Ready to print" is no longer claimed at gate-pass. It is emitted only after successful slice (`apps/ui/src/App.tsx:1268-1362`).
- Send is guarded by fresh slice state before it can proceed (`apps/ui/src/App.tsx:1482-1492`).
- The design-spec right rail exists with Customize, Make it real, VCL status, manual orient controls, printer/material selectors, Slice, Send, and Iteration log (`apps/ui/src/App.tsx:4282-4458`).
- External libraries are no longer raw arbitrary include paths. Admission copies into a sandbox, skips symlinks/hidden paths, caps file count/bytes, and OpenSCAD resolves only approved `library/` or `external/` prefixes (`packages/engine/src/kimcad/external_libraries.py:70-122`, `packages/engine/src/kimcad/openscad_runner.py:119-128`, `packages/engine/src/kimcad/openscad_runner.py:281-296`).
- The public status docs are substantially more honest than before: they state happy-path verification and call out incomplete UI matrix coverage (`docs/STATUS.md:21-33`).

## QA Verdict

The product has made real progress and the core beta happy path is no longer theater. QA would not mark the release gate clean yet because first-run/dependency-absent native behavior and the signature VCL correction loop are not runtime-proven, and the browser matrix is still too thin for a product whose risk is mostly stateful UI and local-tool degradation.

Recommended next action: fix QA-4 and QA-7 immediately, then add the isolated native smoke and VCL correction fixture before treating GauntletGate Full as passable.
