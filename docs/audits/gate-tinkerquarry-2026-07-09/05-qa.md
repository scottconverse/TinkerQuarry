# QA Engineer deep-dive — TinkerQuarry v1.4.0 release gate (2026-07-09)

**Scope:** engine HTTP API/protocol layer, driven directly (no UI), against the
`release/v1.4.0-prep` engine (kimcad `0.9.4`). The runtime walkthrough
(`walkthrough-report.md`) already covered the installed-app UI/first-run flow — this pass
does not repeat that; it exercises the API surface underneath it.

**Method:** started the engine standalone —
`packages/engine/.venv/Scripts/kimcad.exe web --port 8991 --demo --out <temp>` with
`TINKERQUARRY_DEV_TOKEN=tq-gate-qa` set, so every state-changing request carries
`X-KimCad-Session: tq-gate-qa`. Probed with `curl` (PowerShell's `Invoke-WebRequest` and
curl are equivalent for these header/body/status checks; curl gave cleaner raw-header
output for this evidence). Killed the process (PID 15136) when done; confirmed dead
(`ECONNREFUSED` on a final health probe).

## What's working

- **`GET /api/health`** — `200`, `{"version": "0.9.4", "openscad": true, "orcaslicer": true, "cadquery": true, "external_binaries": []}`. Correct: bundled tools resolve, no external-path warning.
- **`GET /api/model-status`** — `200`, honestly reports the demo provider (`"model": "demo", "running": true, "model_present": true, ...`) rather than probing a real Ollama it doesn't need.
- **`POST /api/model-pull` in demo mode** — typed `400` refusal exactly as specified: `{"status": "not_local", "error": "Demo mode doesn't download models — run KimCad without --demo to set up the local AI."}`. Demo mode cannot be tricked into a real multi-GB download. Confirmed.
- **Session-token guard** — a state-changing POST (`/api/reverse-import`) with no `X-KimCad-Session` header is cleanly refused `403 {"error": "Missing or invalid session token. Reload KimCad.", "reason": "session"}`. A GET with no session header still succeeds (by design — GET routes carry no state-changing risk); this matches the code's documented threat model (`webapp.py:1654-1664`).
- **Empty body on `/api/reverse-import`** — `400 {"error": "Empty upload."}`, no hang, no traceback.
- **Malformed non-mesh upload with an unsafe filename** (`evil<>.stl`, plain text bytes) — `200` with an app-level rejection: `{"status": "render_failed", "has_mesh": false, "error": "Could not read that file as a triangle mesh..."}`. The dangerous filename characters are stripped by `re.sub(r"[^A-Za-z0-9._ -]+", "_", ...)` (`reverse_import.py`'s caller in `webapp.py:2676`) before it ever touches the filesystem — confirmed no path-traversal or shell-unsafe name reaches disk.
- **Oversized body (73 MB against a 64 MiB cap)** — fast, bounded `413 {"error": "File too large."}` in 1.3s, not a hang or a connection reset; the declared-length short-circuit (`_read_raw_body`, `webapp.py:3040-3056`) means it never buffers the oversized body at all.
- **Concurrency cap on `/api/reverse-import`** — two simultaneous uploads: one processed (`200`, in 0.17s), the other refused immediately (`429 {"error": "KimCad is busy finishing another design — give it a moment and try again.", "reason": "busy"}` in 0.002s). The `BoundedSemaphore(1)` (`_MAX_INFLIGHT_REVERSE_IMPORTS`) works exactly as designed.
- **Wrong-verb handling on POST-only routes reached via `PUT`/`DELETE`/`PATCH`** (`do_PUT = do_DELETE = do_PATCH = _method_not_allowed`) — correct `405` with a truthful `Allow` header and the JSON `{"error": "Method not allowed."}` envelope.
- **Wrong-verb handling on GET-only routes reached via `POST`/`DELETE`** (`/api/health`, `/api/model-status`) — correct `405`, truthful `Allow: GET, HEAD`, JSON envelope, and (per the code's own comment) evaluated *before* the session-token guard so the error signal is "wrong verb," not a misleading "bad token."

## Findings

### QA-1 · Critical · reverse-import can't recognize its own textbook shapes (bbox-tie-break picks the wrong family, and the algorithm never tries the runner-up)

**Repro:**
1. `trimesh.creation.box(extents=[45,45,45]).export('box.stl')` (a plain solid 45mm cube — the exact case named in this gate's test plan) → `POST /api/reverse-import`.
   - Observed: `{"status":"needs_experimental","error":"Imported mesh matched a known envelope, but its volume or surface area does not match the trusted parametric twin closely enough.","reverse_import":{"matched_family":"snap_box","confidence":1.0,"volume_delta":0.7563,"surface_delta":0.4536,...}}`
2. `trimesh.creation.cylinder(radius=10, height=30).export('dowel.stl')` — a solid cylinder, which is *literally* the shape the shipped `dowel_pin` template family (`packages/engine/src/kimcad/templates.py:2352-2370`, "A solid alignment dowel pin — a plain cylinder") exists to recognize.
   - Observed: **also** rejected, matched to `snap_box` again, `volume_delta: 0.4321`, `surface_delta: 0.5328`.

**Root cause (confirmed by direct call, not inferred):** `match_known_family_from_bbox`
(`packages/engine/src/kimcad/reverse_import.py:40-74`) ranks every registered family purely
by axis-aligned-bounding-box fit and returns a **single** winner; ties are broken by
iteration order over `default_registry().families()` (i.e., by definition order in
`templates.py`'s `_build_default_families`), not by which family's rebuilt geometry
actually resembles the uploaded mesh. I instrumented the matcher directly for the
20×20×30mm cylinder bbox:

```
18 candidates <= tolerance
DOWEL: ('dowel_pin', 0.0, 0.75, {'diameter': 20.0, 'length': 30.0})
best pick (what algo returns): snap_box
```

`dowel_pin` is one of 18 families tied at a perfect bbox score of `0.0` for that envelope
— and it is the ONE among them whose rebuilt geometry would pass the subsequent
volume/surface check (a solid cylinder vs. a solid cylinder). But `snap_box` (a *hollow*
box-like family) is defined earlier in `_build_default_families` and wins the tie
arbitrarily. `_handle_reverse_import` (`webapp.py:2721-2736`) then tests **only** that one
winning family's geometry signature (`geometry_signature_matches`); when it fails, the
whole import is refused as `needs_experimental` — the code never falls through to try any
of the other 17 tied candidates, even though the correct one is sitting right there.

**Why it matters / impact scope:** Reverse import is a documented, shipped v1.x feature
(`docs/USER-MANUAL.md` "Reverse Import Contract", `CHANGELOG.md`, `docs/STATUS.md`:
"Reverse import | Implemented"). Because dozens of templates share simple box-like or
cylinder-like bounding boxes (confirmed: 18-way tie for one modest envelope), **any solid
part whose true family isn't whichever one happens to be registered first** will silently
fail — including a plain block and the cylinder that is the dedicated `dowel_pin` family's
own reference shape. This isn't a synthetic edge case: a plain solid box or cylinder is the
single most likely first thing a real user or evaluator uploads to try this feature, and it
fails both times, with a response (`"confidence": 1.0`) that reads as "we found your part
with perfect confidence" immediately before refusing it — actively misleading about why it
failed. Every family whose bbox can tie with an earlier-registered family in
`_build_default_families` is affected (18 out of the sampled set for one test envelope
alone); this is systemic to the matcher, not a one-off.

**Fix path:** In `match_known_family_from_bbox`, don't return a single bbox winner —
either (a) rank ties/near-ties and let `_handle_reverse_import` try the geometry-signature
check against each candidate in score order until one passes (falling back to
`needs_experimental` only when none do), or (b) fold the volume/surface comparison into the
ranking itself so the "best match" is chosen by full geometry fit, not bbox alone. Either
fix is contained to `reverse_import.py` + the one caller in `webapp.py`.

**Coverage note:** `packages/engine/tests/test_reverse_import.py` (56 lines added this
release) — I did not exhaustively read every case, but given this bug reproduces on the
very first two shapes tried, it is very unlikely the existing suite asserts a positive match
against a shape that collides with an earlier-registered family. Recommend adding this
exact repro (solid cylinder → must resolve to `dowel_pin`, not an earlier-registered
box family) as a regression test.

### QA-2 · Major · GET on a POST-only route returns a bare empty body, breaking the just-fixed JSON error-envelope contract

**Repro:**
```
GET /api/reverse-import   (with a valid session header, so it isn't the token guard)
→ HTTP/1.0 405 Method Not Allowed
  Allow: POST
  Content-Length: 0
  (no Content-Type header, no body)
```
Compare the mirror case, `PUT /api/design`:
```
→ HTTP/1.0 405 Method Not Allowed
  Allow: POST
  Content-Type: application/json
  Content-Length: 32
  {"error": "Method not allowed."}
```

**Root cause:** `do_GET`'s tail (`webapp.py:1327-1334`) has its own inline 405 responder
that was never migrated to call the shared `_method_not_allowed()` helper:
```python
if _is_post_only(self.path):
    self.send_response(405)
    self.send_header("Allow", "POST")
    self.send_header("Content-Length", "0")
    self.end_headers()
    return
```
This is the exact regression the surrounding code says it fixed: `_method_not_allowed`
(`webapp.py:1044-1065`) carries a comment — "QA-006: return the app's JSON error shape (not
an empty body) so the error contract is uniform" — and the single-source-of-truth comment
at `webapp.py:865-867` explicitly says the 405/Allow logic must not "drift between
`_method_not_allowed` and the do_GET/do_POST wrong-verb guards." The routing *tables*
(`_is_get_only`/`_is_post_only`) were unified, but this one response body was not — `do_GET`
still emits the pre-fix empty body while `do_POST`'s equivalent tail-check and all of
`do_PUT`/`do_DELETE`/`do_PATCH` correctly call `_method_not_allowed()` and emit the JSON
envelope.

**Why it matters / impact scope:** Any client — a future SPA change, a monitoring script, an
integrator's tooling, or a test — that relies on the documented/tested JSON error envelope
(`test_webapp.py:806-809` asserts exactly this envelope for the *other* 405 direction) will
get a silent empty body instead when it happens to GET one of the 8 POST-only paths/3
POST-only prefixes. Low likelihood in the shipped SPA (it shouldn't ever GET these), but it
is a real, confirmed contract inconsistency on a public API surface documented in
`packages/engine/docs/api.md`.

**Coverage note:** confirmed by grep — `test_webapp.py` has tests for POST-on-GET-only
(`405` + JSON body, lines ~800-812) and PUT-on-POST-only (`405` + truthful `Allow`, lines
~1848-1854), but no test for GET-on-POST-only. That gap is exactly how this shipped.

**Fix path:** replace the inline block at `webapp.py:1327-1334` with a call to
`self._method_not_allowed()` (it already computes the correct `Allow` for a `_POST_ONLY_PATHS` path); add the missing GET-on-POST-only test alongside the existing mirror-direction tests.
