## 5-line summary

1. **IA core is strong**: describe-first, always-present 3D preview, novice 3-step path, and a single SPA serving both desktop (WebView2) and web — all present and well-built (`Landing.tsx`, `Workspace.tsx`).
2. **Three §8 gaps**: no "CODE IS A DRAWER" affordance at all (generated source never exposed), no user-facing ORIENT step or orient-override in the manufacturing rail (orientation is auto-only, a read-only chip), and the rail is split across two Inspector tabs rather than one continuous Readiness→Orient→Slice→Print path.
3. **Two §9/§7.18 state gaps**: there is **no app-level OFFLINE banner** (offline only shows as a per-printer connector status) and **no friendly CRASH/RECOVERY screen** (no React error boundary — a render exception white-screens); override framing is "proceed with caution," softer than the PRD's "overriding a safety check."
4. **§11 API contract is ~90% covered**: every design status + progress phase, send/simulated, print-outcome, printer status, model status, and template tier are handled with distinct UI; the only weak spot is **slice not-sliced reasons** (`no_profile`/`failed`/`stale`) which collapse into one generic `slice.note` message with no `stale`-specific path.
5. **§12 security and privacy are excellent and real**: per-boot session token (constant-time, Sec-Fetch-Site for GETs, reload recovery), loopback default with warned `--allow-remote`, OpenRouter key in OS keyring (masked-only), SCAD sanitizer + arm's-length worker, and **no telemetry/egress in app source** (only a transitive lockfile dep) — the one reliability caveat is the missing UI error boundary.
