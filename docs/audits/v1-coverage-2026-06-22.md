## 3. Net-new findings (not in prior docs/audits or the owner's auditor's list)

1. **Phantom citations in the requirements map itself.** The §6.10 maturity-state rows cite `apps/ui/src/components/SendPanel.tsx:182/194/255/258` etc. as evidence of "built-automated-tested" states. **That file does not exist** (`find` returns nothing; `engine.send`/`engine.outcome` have 0 callers). Any reader trusting those rows would over-credit the send surface. The honest reading collapses all six "maturity state" rows into the single fact: engine + `engineClient.ts` methods exist, **no send UI ships**. The prior honesty-audit said the loop is missing but did **not** flag that the map's own evidence lines point at a non-existent file.

2. **Generated design code is genuinely unreachable in the UI — confirmed at the wiring level.** Prior audits said "code view exists for the user's own files." Stronger finding: `engine.source()` (`engineClient.ts:168`) — the one method that fetches the *generated* SCAD — is called only by `EngineStatusBanner.tsx` and `WelcomeScreen.tsx`, never by `App.tsx` to populate Monaco. So §7.7 "show me the code" for an AI-designed part is not merely "collapsed by default," it is **not plumbed at all**.

3. **The offscreen multi-view renderer is built and working but orphaned.** `offscreenRenderer.ts` (7 preset views) is real and shipping via ShareDialog — yet it is the *exact* input the (missing) Visual Correction Loop would consume. The owner is paying the maintenance cost of half the loop's plumbing while getting none of the loop's value. No prior doc framed it as "the capture half exists, the brain half is zero."

4. **First-run tool-health is a confirmed regression against §7.1, not just "needs verification."** The map hedged ("requires verification"). Verified: `FirstRunWizard.tsx` never imports `getHealth`; a first-run user with no OpenSCAD/slicer gets **no guidance until they design and fail**. The detection code exists but is buried in Settings.

5. **"Version history" is doubly weaker than stated.** It is not only in-session (`VersionRail` = React state) — `design_store.py:32` persists exactly one snapshot and **overwrites it on save**, and `App.tsx:339` implements *destructive* forward-history truncation. A reopen loses all prior versions. The §6.12 "every meaningful iteration retained" requirement is contradicted by the storage layer, not merely unfinished.

## 4. Blunt verdict

**The product is not close to v1, and the signature value is absent.** What is real is the *factory floor*: the engine slices real motion-bearing G-code, the gate refuses bad parts server-side fail-closed, save/reopen/source round-trips under real HTTP tests, security (per-boot token, SCAD sandbox, loopback-only) is genuinely tested, and the Studio-inherited 3D viewer is solid. That is a credible, trustworthy CAD-to-G-code engine. But TinkerQuarry's *reason to exist* — an AI that **looks at the render and catches spatial errors math can't** — is zero code, its acceptance test (wrong-face hole) **cannot be run**, the local vision model already failed it 0/3, and the punted "cloud-optional" path was never written. Below that, the "make it real" arc dead-ends at a file download: there is **no send-to-printer UI and no post-print-outcome prompt** despite tested engine endpoints sitting unused. And the entire front-end is structurally unverified — **no `App.test.tsx`, no test that POSTs a real design and renders it** — which is the precise gap that let the §6.12 reopen bug ship before. Call it honestly: a strong v1 *engine* wearing a v0.6 *application*, missing its one differentiating feature.
