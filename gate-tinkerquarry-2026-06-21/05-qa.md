## Findings

### QA-1 (Major) тАФ `POST /api/settings` silently accepts unknown/typo'd fields and returns `saved:true`
**Evidence (request тЖТ response):**
```
POST /api/settings  X-KimCad-Session: <valid>   body {"bogus_field":"x"}
тЖТ 200  {... "saved": true}
```
**Observed vs expected (api.md ┬зSettings, line 181):** api.md states *"POST merges any of those fields (validated; unknown values are a `400`, a typo'd field is an error...)"*. The real handler (`webapp.py:_handle_settings_post`, lines 1725-1804) validates only the **values** of *known* keys; an unrecognized **field name** is never inspected тАФ it falls through, `updates` stays empty, and the server returns `200 {"saved": true}`. The sibling handler `_handle_connections_post` (line 1578) *does* reject unknown fields with a `400` (`"Unknown connection field(s): ..."`), so the contract is internally inconsistent.
**Why it matters:** The `saved` flag is explicitly documented as "honest about whether the write stuck." A user (or integrator) who typos `experimental_enable` (missing `d`), `default_printr`, or `cloud_modle` gets `saved:true` while the setting silently no-ops тАФ a false-positive honesty signal and silent config-intent loss. This is a contract/honesty deviation, not a security hole (no unsafe value is stored).
**Blast radius:** Settings screen + any API client; data-integrity/honesty, not safety. Low exploitability.
**Fix:** Mirror the connections handler тАФ compute `unknown = set(data) - KNOWN_SETTINGS_FIELDS - {"reset"}` and return `400 {"error": "Unknown settings field(s): ..."}` before merging. Or, if silent-ignore is the intended posture, soften api.md line 181 to match.

### QA-2 (Minor) тАФ `print-outcome` on an unknown rid returns 404, not the documented 409
**Evidence:**
```
POST /api/print-outcome/999  {"outcome":"clean"}  тЖТ 404 {"error":"That design is no longer available."}
```
**Observed vs expected (api.md ┬зprint-outcome, lines 130-131):** api.md documents the gating failure as `409 {"error":"Record an outcome after a real printer send."}`. For a *known* design that was never really sent, the code (lines 1943-1951) does return that 409 тАФ but an *unknown* rid hits the `snap is None` 404 branch first. So the 409 path is reachable only for a rid that exists in the registry. This is defensible (truly-unknown id is genuinely a 404), but a client probing the "no real send" guard with an arbitrary id sees 404 where the doc implies 409.
**Why it matters:** Cosmetic/contract precision; the *safety* invariant (no outcome recorded without a real non-simulated send) holds either way тАФ both branches refuse. No data is recorded.
**Blast radius:** API clients relying on exact status codes. Negligible.
**Fix:** Either note in api.md that an unknown id is a 404 (the 409 is specifically "known design, no real send"), or order the `can_record` check first for a uniform 409. Doc tweak preferred.

### QA-3 (Nit) тАФ `/api/health?recheck=1` cross-origin returns 200 (cached), api.md says "refuse"
**Evidence:**
```
GET /api/health?recheck=1   Sec-Fetch-Site: cross-site  тЖТ 200 {"version":"0.9.3", ...}
```
**Observed vs expected (api.md security summary, lines 248-250):** api.md says the health re-probe "refuse[s] a cross-origin request via the browser's `Sec-Fetch-Site` header." The implementation (lines 990-1002) is subtler and *more correct than the doc reads*: it **skips the CPU-bound CadQuery re-probe** for a cross-origin caller (the actual side effect a malicious page could weaponize) but still answers with **cached** health (a harmless read). The security-relevant invariant тАФ no cross-origin-triggered repeated probe тАФ holds. `/api/step/<id>` (a real side-effecting build) *does* hard-403 cross-origin, verified. Only the doc wording is looser than the code.
**Fix:** Reword api.md to "skips the side-effecting re-probe for a cross-origin request (still returns cached health)," distinguishing it from `/api/step` which is hard-refused.

## Probes that PASSED (the invariants this lane was sent to break)

**Session-token protocol (api.md KC-26) тАФ solid.** Token scraped from `<meta name="kimcad-session-token">` in the page shell. Every state-changing POST is gated:
- `/api/design`, `/api/slice/<rid>`, `/api/send/<rid>`, `/api/print-outcome/<rid>`, `/api/settings`, `/api/render/<rid>`, `/api/connections`, `/api/model-pull` тАФ **all 403 without the header**, with the typed body `{"error":"Missing or invalid session token. Reload KimCad.","reason":"session"}` (the `reason:"session"` the SPA uses for its Reload banner).
- **Wrong token тЖТ 403**, **empty-string token тЖТ 403** (constant-time `hmac.compare_digest`, `webapp.py:1399`).
- **WITH the page-shell token тЖТ request is processed (200)** тАФ confirmed end-to-end on `/api/design` (took 166s cold; pipeline returned a typed `model_unavailable` body, proving the request passed the guard and reached the pipeline).
- No state-changing route bypasses the guard. Wrong-verb POST to a GET-only route (`/api/health`, `/api/options`) correctly returns **405 before** the token check (so integrators get the truer "wrong method" signal) тАФ these are read-only so nothing is weakened.

**Side-effecting GET gating.** `/api/step/<id>` cross-origin тЖТ **403** (refuses a drive-by CadQuery build). Ordinary GETs (`/api/options`) never gated. (See QA-3 for the health re-probe nuance.)

**Safety invariants тАФ cannot be broken via direct API:**
- `slice` on a non-existent design тЖТ **404** ("Design the part first"). `send` on a never-sliced part тЖТ **404** ("Slice the part first"). `print-outcome` without a real send тЖТ **404/409** (refused; nothing recorded).
- Gate-failed parts: code fails **closed** тАФ `reg.gate_status[rid] = rep.get("gate_status") or "fail"` (webapp.py:2158); both `_handle_slice` (2480) and `_handle_send` (1872) return `{"sliced":false,"reason":"gate_failed"}` / `{"sent":false,"reason":"gate_failed"}` server-side regardless of client claims (belt-and-suspenders: send re-checks even if a gcode entry exists).
- `send` records a real-send only when `simulated` is false (`not connector.drives_hardware`, line 1919) тАФ a simulated `mock` send never unlocks `print-outcome`. Confirmed `connector-status/mock` тЖТ `simulated:true`, `bambu_p2s` тЖТ `simulated:false, reason:config`.

**Pipeline robustness.** empty / whitespace / missing prompt тЖТ **400** "Please describe the part you want." (validated before any model call). Non-object body тЖТ 400; malformed JSON тЖТ 400; body >1 MiB тЖТ **413** (drained); 2nd concurrent design тЖТ **429** busy (single in-flight slot, `BoundedSemaphore`). All recoverable conditions are typed 200/4xx, **never a 500** тАФ matches api.md's "recoverable condition is never a 500."

**Mock vs real parity.** `tinkerquarry/backend/mock_api.py` encodes the same status contract (gate-passтЖТsliceтЖТconfirm-sendтЖТreal-send-gated-outcome) and the same `simulated` semantics. Real server correctly **omits** the mock's wildcard `Access-Control-Allow-Origin: *` (a dev affordance) and relies on token + `Sec-Fetch-Site` instead тАФ the stronger posture. Mock's `print-outcome` returns 409 for a non-real-sent *known* rid, matching the real server's known-rid path. One drift: the mock returns `422` for a bad outcome value where the real server returns `400` (`"Unknown print outcome."`) тАФ minor and on the mock's side, not the shipping product.

**CLI parity.** `kimcad design <prompt> [--slice] [--send CONNECTOR] [--proceed-anyway]` exposes the same confirmation gates as the web path: `--slice` = explicit slice confirmation, `--send` = slice-then-send, `--proceed-anyway` = the documented gate override. Prior `output/tq-real-cli` artifact proves end-to-end completion. Structurally parallel to the web pipeline; gate stance ("download to inspect, never auto-send") mirrors the web `gate_failed` refusal.
