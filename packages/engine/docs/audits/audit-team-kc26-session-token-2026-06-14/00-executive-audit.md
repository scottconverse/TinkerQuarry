# Executive Audit — #31 / KC-26 session-token guard
**Date:** 2026-06-14 · **Scope:** the session-token change (commits 43a0c96 + 9435931), 5 roles, balanced, audit-only.
**Method:** 5 independent role audits run in parallel (Principal Engineer, QA, Test Engineer, Technical Writer, UI/UX) via a workflow with structured findings.

## Executive summary
The server-side guard is **sound and complete**: a single `hmac.compare_digest` check at the top of `do_POST` covers all 13 state-changing routes, the per-boot `secrets.token_urlsafe(32)` token is generated by both production server-start sites (`webapp.serve()` and the WebView2 `shell.py`), the SPA stamps it on exactly the POSTs, GETs are never gated, and the token never leaks operationally. No Blocker or Critical. The roles converged on one real gap the original change underweighted — **client-side recovery**: the SPA read the token once at module load, the token rotates per boot, so a stale tab after a restart 403s every action with no recovery path (autosave looped forever; 7 of 9 surfaces showed misleading domain copy). Two side-effecting **GETs** (the lazy STEP build, the health re-probe) also bypassed the POST-only guard, and the test/doc coverage was thinner than the docs claimed. **All 24 findings were remediated in this pass and re-verified.**

## Findings + remediation (by theme)

### Theme A — Session-expiry recovery (QA + UI/UX, 5 Major + 2 Minor) — REMEDIATED
The dominant finding. The SPA read `SESSION_TOKEN` once at module load (`api.ts`); the token rotates per boot (`serve()`/`shell.py`). A stale tab after a restart → every POST 403s; autosave (`App.tsx`) re-fired the doomed request every 1.5s forever; Send/Settings/Connections/rename/delete showed misleading domain copy ("couldn't reach the printer", "didn't stick — try again") instead of "reload"; the WebView2 shell has no refresh chrome, so even the "Reload KimCad" string was undiscoverable.
**Fix:** the 403 now carries a typed `reason:"session"`; `api.ts` throws a typed `SessionExpiredError` and fires an app-registered handler; `App.tsx` shows a prominent, non-dismissable **Reload banner** (one-click `location.reload()`, which re-fetches the freshly injected token — guaranteed recovery, works in WebView2); the autosave retry loop is stopped on a session-403. Pinned by `App.test.tsx` (banner renders on the handler) + `api.session-token.test.ts` (typed error + handler fires; plain 403 stays ordinary).

### Theme B — Side-effecting GETs bypass the POST guard (Principal Engineer Major + Minor) — REMEDIATED
`GET /api/step/<id>` lazily spawns a CadQuery build + writes files; `GET /api/health?recheck=1` re-runs the CadQuery probe. Both are reads/navigations that can't carry the token, so the `do_POST` guard never covered them — a cross-origin drive-by could trigger builds/probes (bounded resource abuse).
**Fix:** a `_is_cross_site()` check (`Sec-Fetch-Site: cross-site/cross-origin`) refuses the STEP GET and skips the health re-probe for a cross-origin request; same-origin and non-browser/test clients are unaffected (fail-open). Pinned by `test_cross_origin_get_cannot_trigger_side_effecting_builds_or_reprobes`.

### Theme C — Test breadth (Test Engineer, 2 Major + 3 Minor/Nit) — REMEDIATED
The 403/route-through was proven for one route (`/api/settings`) and the frontend header only via `postDesign`.
**Fix:** the backend guard test now asserts a tokenless 403 across a representative spread (`/api/design`, `/api/slice/1`, `/api/designs/import`, `/api/model-pull`, `/api/connections`, `/api/settings`) and asserts `== 200` (not `!= 403`) for the route-through; the frontend test now exercises the postJson path (`postSettings`), the no-headers path (`startModelPull`), and a raw-body upload (`importDesign`), reading the header robustly via `new Headers()`. `apiFetch` now normalizes all header forms via `new Headers()` (was a record-only spread, a latent footgun).

### Theme D — Docs accuracy/completeness (Writer 1 Major + Minors, UI/UX Nit) — REMEDIATED
- `api.md` advertised `GET/DELETE /api/designs/<id>` though no DELETE exists (delete is `POST .../delete`) → **fixed** the heading + added a body note.
- `SECURITY.md` omitted the token though `api.md` points there for "full detail" → **added** the session-token section.
- No user-facing doc explained the 403 "Reload KimCad" symptom → **added** a `troubleshooting.md` entry.
- The CORS-preflight rationale was over-weighted vs the real (unreadable-token) defense, and the token's non-authentication of `--allow-remote` was unstated → **reordered + clarified** in `api.md`/`SECURITY.md`, and the `cli.py` "NO authentication" warning now explicitly notes the token is anti-cross-origin only.

### Nits — REMEDIATED
- `static_cache` lock-free under ThreadingHTTPServer → documented as intentional (idempotent writes; benign GIL-atomic race).
- Constant-time property: covered by the docstring rationale; threat is theoretical (local single-user token).

## What's working (credited by the roles)
- The guard is structurally complete: one top-of-`do_POST` check covers all 13 state-changing routes; no `do_PUT`/`do_DELETE`/`do_PATCH` mutate; only `serve()` + `shell.py` construct the SPA handler, both with a token.
- 256-bit per-boot CSPRNG token, never persisted, not echoed in any response/header/log (`log_message` silenced).
- The token-keyed index-shell ETag cache cannot serve a wrong-token body; build reproducibility preserved (only the placeholder is committed).
- The off-by-default-in-tests contract is honest and pinned, so the ~135 existing POST tests are genuinely unaffected.
