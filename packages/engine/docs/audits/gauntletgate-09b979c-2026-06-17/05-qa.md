## Findings

### QA-GG-001 — Managed `ollama serve` child is orphaned on app exit (corroborates ENG-GG-001) — **CRITICAL**
**Evidence.** `src/kimcad/ollama_runtime.py:12` docstring promises the managed runtime is "start[ed] … as a **managed subprocess** … and stop[ped] … with the app." Runtime/lifecycle inspection (`grep -nE "atexit|terminate|\.kill|JobObject|stop"` over `ollama_runtime.py`) finds **no teardown** for the spawned `serve` — only `creationflags=CREATE_NO_WINDOW` (`ollama_runtime.py:149`), which hides the console, not the lifecycle. `serve()` calls `ensure_serving_background()` (`webapp.py:2747-2749`, `shell.py:119-121`) but the server never registers a stop. On Windows a bare `subprocess.Popen` child is not killed when the parent exits (no job object), so on a cold-start machine — the exact target of this commit — closing KimCad leaves `ollama serve` resident (multi-GB for the portable build) holding `:11434`.
**Why I could not runtime-repro the orphan here (honest):** the dev box already has a system Ollama running, so `is_up()` short-circuits before any spawn — which is precisely why this is invisible to a warm-box walkthrough. I confirm the *defect* by code+contract; the engineering lane owns the primary write-up (ENG-GG-001) and the same root cause.
**Blast radius.** Every cold-start (or system-Ollama-installed-but-stopped) machine; a leaked background server across reboots, masked on the next launch by `is_up()`. Directly contradicts two shipped docstrings.
**Fix.** Track the spawned `Popen` and tear it down: `atexit`-register a `terminate()` (and on Windows assign the child to a Job Object with `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`, or pass `CREATE_NEW_PROCESS_GROUP` and signal it) in `serve()`'s `finally`. Only stop a server KimCad itself started (never a pre-existing system one).
**Suggested test:** a `real_tool` test that, with no system Ollama, starts the app, asserts a managed `serve` came up, exits the app, and asserts the child PID is gone within N seconds.

### QA-GG-002 — Stated part dimensions are silently dropped: an "8 mm cable" yields a 6 mm clip — **MAJOR**
**Evidence.** Two real runs (one over the API, one in-process) of `"a desk cable clip for an 8 mm cable"`:
- API payload `parameters` → `cable_d` `value:6.0` (the template default), not 8.
- In-process `pipeline.run(...)` → `plan.dimensions == {'width':16.0,'height':23.4,'cable_d':6.0}` and `template.values == {'cable_d':6.0,'width':16.0}`.
The `cable_clip` family maps `cable_d` from `dim_keys=("cable_d","cable_diameter","diameter")` with `default=6.0` (`templates.py:519`). The planner (qwen2.5:7b) did not emit the user's stated 8 mm into any of those keys, so the template fell back to its default. The part is printable and the gate passes — but it is **not the part the user asked for**. KimCad's headline promise is "describe a part, get *that* part"; a silently-substituted cable diameter erodes exactly that trust, and nothing in the UI flags that a stated dimension was ignored.
**Why Major, not Critical:** the part still renders, passes the gate, and the live slider lets the user correct `cable_d` to 8 — so it is a fidelity/accuracy gap on a recoverable flow, not a broken feature. But it is reproducible on the flagship example prompt and is a runtime-only finding (no other lane exercised the warm model).
**Fix.** Strengthen the plan→template dimension mapping: (a) extend `dim_keys` / the prompt schema so a "N mm cable" reliably lands in `cable_d`; (b) when the user's prompt contains an explicit dimension the plan failed to bind, surface a one-line "I used the default 6 mm channel — adjust below" note rather than silently defaulting. Consider a planner prompt tweak that echoes parsed dimensions back for binding.
**Suggested test:** a `real_tool` (or seeded-plan) assertion that a prompt naming a cable diameter produces `cable_d` matching it (within tolerance), or else an explicit "defaulted" flag on the payload.

### QA-GG-003 — `/api/mesh/1` returned 404 once on the very first design after boot (observed, NOT reproducible) — **MINOR**
**Evidence.** In the first warm run, the cable_clip design (rid=1) returned `has_mesh:true` + `mesh_url:/api/mesh/1`, yet `GET /api/mesh/1` → `404 {"error":"mesh not found"}` and `output/web/1/` held only the `part-v0.*` STEP-build artifacts, no `part.oriented.stl`. I could **not reproduce** it: rid=2 (box) and rid=4 (cable_clip) both served their mesh (HTTP 200, correct bytes) in the same session, and a clean restart with the cable_clip as the first design produced `part.oriented.stl` (15,484 B) and `/api/mesh/1` → 200. The STEP lazy-build was cleared as the cause (mesh and `part-v0.*` coexist; mesh re-fetch stays 200 after a STEP build).
**Why Minor (not higher):** a 404 here means the 3D preview is blank for that design — a real UX failure if it recurs — but it occurred exactly once in ~6 designs and survived no reproduction attempt, so I will not inflate it. Recording it honestly so it isn't lost: a first-design-after-boot transient (possibly a `DesignRegistry.__init__` startup cleanup `shutil.rmtree` racing the first render, `design_registry.py:45-47`) is the leading hypothesis and is worth a defensive look.
**Fix path.** Audit the startup `rmtree` of `web_root/<digit>` dirs vs. the first design's render timing under the real (slower) interpreter import; consider deferring/guarding the cleanup so it can never race a concurrent first render.
**Suggested test:** a stress test that fires the first `/api/design` immediately after `serve()` boot and asserts the returned `mesh_url` resolves 200.

### QA-GG-004 — Warm `/api/settings` reflects the real user profile, not an isolated one (runtime hygiene note) — **MINOR**
**Evidence.** The warm run launched from the repo with the real `USERPROFILE`, so `/api/settings` returned the developer's actual `cloud_enabled:true` + connector config, and render artifacts landed in the repo's CWD-relative `output/web/` (`paths.output_dir()` → `…\kimcadclaude\output`). That is correct *product* behavior (a real user runs warm against their own profile), and the local provider was still used (no cloud key). The note: the warm lane has no profile isolation, so a tester must be careful not to conflate dev-profile state with shipped defaults — the Walkthrough lane's cold run is the one that proved isolation. No code defect; flagged so the gate's evidence chain is unambiguous.
**Fix path.** None required for the product. For test hygiene, document that warm-lane runs touch the real `~/.kimcad`/`output` and clean up (I removed `output/web/` after the run).

## What's working (verified, honest signal)

- **The core promise, end to end on real tools.** Live qwen2.5:7b plan → real OpenSCAD render → gate PASS → real OrcaSlicer `.3mf` with 10,445 G1 motion moves, inspected losslessly. This is the single most important thing the gate must confirm, and it holds.
- **Warm `/api/model-status` is honest and complete** (`webapp.py:1683-1709`): loopback-host classification (the ENG-COLD-002 fix) means the local Ollama is *probed*, not assumed — `running:true`, `model_present:true`, AND `vision_present:true` all reflect a real probe of `:11434`, not a substring guess.
- **Per-boot session-token guard is live and correct.** `POST /api/design`, `/api/slice/1`, `/api/settings`, `/api/render/1` without the token → **403** `{"reason":"session"}`; with the token from the index shell's `<meta name="kimcad-session-token">`, the design POST succeeds. Constant-time compare (`webapp.py:1395`).
- **Server-side gate-failure refusal** (`webapp.py:2476-2480`): a part that fails the printability gate is refused at `/api/slice` with `reason:gate_failed`, mesh still downloadable — proven live with a 500 mm cube.
- **Protocol hygiene:** wrong-verb `DELETE /api/health` → **405** with truthful `Allow: GET, HEAD`; an oversized (>1 MiB) `/api/design` body → **413**. The single-source route table (`webapp.py:684-696`) backs these.
- **`ollama_fetch.py` integrity path is real, not a string assertion** (`ollama_fetch.py:86-132`): SHA-256 is computed while streaming and checked **before** any extraction; zip-slip guarded by `resolve()` + `relative_to(dest)` (`_safe_extract`, lines 47-65); streamed to a temp file (no 1.4 GB in RAM) with `finally` cleanup; version+hash pinned together. The 44 new-module unit tests pass (`test_ollama_runtime/fetch/model_pull/cadquery_worker`), and `test_webapp.py` + `test_ollama_runtime_real.py` = **144 passed**.
- **`cadquery_worker._deny_network`** (`cadquery_worker.py:123-133`) raises `PermissionError` on socket construction; exercised live via the rid=4 STEP build (valid 27 KB STEP out). The docstring's honest layer-division (sanitizer = primary, restricted-builtins + deny-network = defense in depth) matches the code.
- **`/api/health` installed-mode pathing** resolves all three bundled binaries (openscad/orcaslicer/cadquery true).
